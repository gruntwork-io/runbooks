The `<Template>` block generates files from a [Gruntwork Boilerplate](/authoring/boilerplate/) template directory. It renders a form for any variables defined in the template, and saves generated files to the workspace.

## Basic Usage

```mdx
<Template id="vpc-setup" path="templates/vpc" />
```

This loads the `boilerplate.yml` file from `templates/vpc/boilerplate.yml` (relative to your runbook file), renders a form for the variables, and generates files when the user clicks "Generate".

## vs. TemplateInline

Template and [TemplateInline](/authoring/blocks/templateinline/) both render Boilerplate templates, but they serve different purposes:

| Feature | `<Template>` | `<TemplateInline>` |
|---------|-------------|-------------------|
| Template source | Directory with `boilerplate.yml` | Inline in runbook |
| Form rendered | Yes, from `boilerplate.yml` | No form (uses [Inputs](/authoring/blocks/inputs/)) |
| File generation | Always saves to workspace | Optional (`generateFile={true}`) |
| Use case | Generate files from template directories | Show preview of single files inline |

Use `<Template>` when you have a full Boilerplate template directory with multiple files and a `boilerplate.yml` configuration. Use `<TemplateInline>` when you want to show a quick inline preview or generate a single file without a separate template directory.

## Props

### Required Props

- `id` (string) - Unique identifier for this component. Used to reference this Template's variables from other blocks.
- `path` (string) - Path to the boilerplate template directory, relative to the runbook file. The directory must contain a `boilerplate.yml` file.

### Optional Props

- `inputsId` (string | string[]) - ID of Inputs or Template block(s) to import variable values from. When multiple IDs are provided as an array, variables are merged in order (later IDs override earlier ones).

## Directory Structure

The `path` prop should point to a directory containing a `boilerplate.yml` file and any template files:

```mdx
<Template id="my-template" path="templates/vpc" />
```

Expected directory structure:

```
templates/vpc/
├── boilerplate.yml    # Variable definitions (required)
├── arbitrary_template_files.txt            # Template file
```

The `boilerplate.yml` file defines the variables that will be collected from the user. Template files use [Boilerplate's Go template syntax](/authoring/boilerplate/#template-syntax) for variable substitution.

## Remote Templates

Templates can be loaded from remote sources instead of local directories. Remote templates are downloaded when the runbook loads, with the Template block showing a "Downloading remote template..." status while the rest of the runbook renders normally.

### Supported Remote Sources

Runbooks supports OpenTofu/Terraform-style module source syntax:

| Source Type | Format | Example |
|-------------|--------|---------|
| **GitHub** (shorthand) | `github.com/org/repo//path` | `github.com/acme/templates//vpc?ref=v1.0` |
| **GitLab** (shorthand) | `gitlab.com/org/repo//path` | `gitlab.com/acme/templates//vpc` |
| **Bitbucket** (shorthand) | `bitbucket.org/org/repo//path` | `bitbucket.org/acme/templates//vpc` |
| **Git** (explicit) | `git::URL` | `git::https://example.com/repo.git//path` |
| **HTTP** (direct download) | `https://URL` | `https://example.com/template.tar.gz` |
| **S3** | `s3::URL` | `s3::https://s3.amazonaws.com/bucket/path` |

### GitHub Shorthand (Recommended)

The simplest way to reference GitHub templates—just use `github.com/...` directly:

```mdx
<Template 
  id="vpc" 
  path="github.com/gruntwork-io/terragrunt-scale-catalog//templates/boilerplate/aws/github/account?ref=v1.3.2" 
/>
```

The URL structure follows [OpenTofu/Terraform conventions](https://opentofu.org/docs/language/modules/sources/):
- Use `//` (double slash) to separate the repo from the subdirectory path
- Use `?ref=` to specify a branch, tag, or commit

### Explicit Git Protocol

For non-GitHub hosts or when you need explicit control, use the `git::` prefix:

```mdx
{/* HTTPS */}
<Template 
  id="vpc" 
  path="git::https://git.example.com/org/templates.git//vpc?ref=v1.0.0" 
/>

{/* SSH */}
<Template 
  id="vpc" 
  path="git::git@git.example.com:org/templates.git//vpc?ref=main" 
/>
```

### HTTP Direct Download

For templates served as archives over HTTP(S):

```mdx
<Template 
  id="vpc" 
  path="https://example.com/templates/vpc.tar.gz" 
/>
```

<Aside type="tip">
All of these formats work for GitHub/GitLab/Bitbucket repos:

```mdx
{/* All equivalent - pick your preferred style */}
<Template path="github.com/org/repo//templates/vpc" />
<Template path="https://github.com/org/repo//templates/vpc" />
<Template path="git::https://github.com/org/repo//templates/vpc" />
```

Runbooks auto-converts these to the Git protocol format internally.
</Aside>

<Aside type="caution">
Remote templates require network access. If you're running Runbooks in an air-gapped environment, use local templates instead.
</Aside>

## Template Dependencies

Boilerplate templates can declare [dependencies](https://github.com/gruntwork-io/boilerplate#dependencies)—nested templates that are rendered as part of the parent template. When a template has dependencies, Runbooks automatically discovers and loads variables from all dependency templates.

### How It Works

When you use a Template block, Runbooks:

1. Loads the main template's `boilerplate.yml`
2. Checks for any `dependencies` declared in that file
3. Recursively loads each dependency's `boilerplate.yml`
4. Collects the **union of all variables** from the main template and its dependencies
5. Displays all variables in a single form

This means users see all required fields—even those defined in nested dependency templates—without you having to manually declare them.

### Example

Consider a template with this structure:

```yaml
# templates/aws-setup/boilerplate.yml
variables:
  - name: AccountName
    type: string
  - name: Environment
    type: enum
    options: [dev, prod]

dependencies:
  - name: aws-account
    template-url: ../shared/aws-account
    output-folder: .
```

And the dependency template:

```yaml
# templates/shared/aws-account/boilerplate.yml
variables:
  - name: AWSAccountID
    type: string
  - name: AWSRegion
    type: string
    default: us-east-1
```

When you use `<Template path="templates/aws-setup" />`, the form will show **all four variables**:
- `AccountName` (from main template)
- `Environment` (from main template)
- `AWSAccountID` (from dependency)
- `AWSRegion` (from dependency)

<Aside type="note">
If the same variable name appears in both the parent and a dependency template, the parent template's definition takes precedence.
</Aside>

### Conditional Dependencies

Boilerplate supports conditional dependencies using the `skip` field:

```yaml
dependencies:
  - name: python-source
    template-url: "../lambda-src/python"
    skip: '{{ not (hasPrefix "python" .Runtime) }}'
    
  - name: nodejs-source
    template-url: "../lambda-src/nodejs"
    skip: '{{ not (hasPrefix "nodejs" .Runtime) }}'
```

**How Runbooks handles conditional dependencies:**

- **Form display**: Variables from *all* dependencies are shown, regardless of skip conditions. Since variable values aren't known until the user fills in the form, Runbooks can't evaluate which dependencies will be skipped.
- **Template generation**: When the user clicks "Generate", the boilerplate library evaluates skip conditions with the actual variable values and only processes the relevant dependencies.

This means some form fields may be for dependencies that won't actually be used. This is intentional—it ensures users can provide values for any possible code path.

<Aside type="tip">
If a dependency's variables have sensible defaults, they won't block generation even if that dependency ends up being skipped.
</Aside>

## With Variables

There are several ways to provide variables to a Template.

### Standalone Template

The Template displays a form with all variables defined in its `boilerplate.yml`:

```mdx
<Template id="vpc-setup" path="templates/vpc" />
```

### Using inputsId

Import variables from a separate Inputs block:

`````mdx
<Inputs id="config">
```yaml
variables:
  - name: Environment
    type: enum
    options: [dev, staging, prod]
    default: dev
  - name: Region
    type: string
    default: us-east-1
```
</Inputs>

<Template id="vpc" path="templates/vpc" inputsId="config" />
`````

Variables from `config` are merged with the template's own variables. Shared variables become read-only in the Template form.

### Using Multiple inputsIds

You can reference multiple Inputs blocks by passing an array of IDs. Variables are merged in order, with later IDs overriding earlier ones:

`````mdx
<Inputs id="org-config">
```yaml
variables:
  - name: OrgName
    type: string
```
</Inputs>

<Inputs id="env-config">
```yaml
variables:
  - name: Environment
    type: enum
    options: [dev, prod]
```
</Inputs>

<Template 
  id="infrastructure" 
  path="templates/infra" 
  inputsId={["org-config", "env-config"]} 
/>
`````

## Variable Categories

When a Template references external inputs via `inputsId`, variables fall into three categories:

| Category | Description | Form Behavior |
|----------|-------------|---------------|
| **Local-only** | Exists only in the template's `boilerplate.yml` | Editable |
| **Imported-only** | Exists only in imported sources | Not shown, but passed to template |
| **Shared** | Exists in both template and imported sources | Read-only, live-synced |

<Aside type="note">
Shared variables are displayed in the form but are read-only and stay synchronized with the values from the imported sources.
</Aside>

## Template File Syntax

Template files use [Boilerplate's Go template syntax](/authoring/boilerplate/#template-syntax):

```hcl
# main.tf
resource "aws_vpc" "main" {
  cidr_block = "{{ .CidrBlock }}"
  
  tags = {
    Name        = "{{ .VpcName }}"
    Environment = "{{ .Environment }}"
  }
}

{{- if .EnableFlowLogs }}
resource "aws_flow_log" "main" {
  vpc_id = aws_vpc.main.id
  # ...
}
{{- end }}
```

## Common Use Cases

The `<Template>` block works especially well for generating files from structured templates:

- **Generate infrastructure code**: Create Terraform, Terragrunt, or other IaC files based on user inputs.
- **Scaffold projects**: Generate project structures with configuration files customized to user preferences.
- **Create configuration files**: Generate CI/CD pipelines, Docker Compose files, Kubernetes manifests, or other config files.
- **Multi-file generation**: Generate multiple related files that need consistent variable values.

## Complete Example

Here's a complete runbook showing Template with imported variables and validation:

`````mdx
# Deploy a VPC

First, configure your environment:

<Inputs id="env-config">
```yaml
variables:
  - name: Environment
    type: enum
    options: [dev, staging, prod]
    default: dev
  - name: Region
    type: string
    default: us-east-1
```
</Inputs>

Now configure your VPC. The Environment and Region will be inherited from above:

<Template id="vpc-setup" path="templates/vpc" inputsId="env-config" />

/>
`````
