## You shouldn't be the bottleneck

You're the person everyone depends on for infrastructure. The Slack DMs pile up. The tickets stack up. You've tried writing docs, but they go stale. You've tried developer portals, but they took months to set up and developers barely use them.

**There's a better way.** Runbooks are live, interactive documents that guide developers through complex tasks step-by-step — with forms to collect inputs, checks to validate prerequisites, and commands that execute with the right parameters. It's developer self-service that developers actually use.

## How it works

You write a runbook in MDX — markdown with interactive blocks — in your editor, just like code. A developer opens it in their browser and gets a guided experience: fill out forms, run checks, execute commands, generate files. No guesswork, no copy-paste errors, no Slack DMs to you.

```mdx
# Deploy a New Service

## Prerequisites

<Check
    id="check-kubectl"
    command="kubectl version"
    title="Check if kubectl is installed"
    successMessage="kubectl is installed!"
    failMessage="Please install kubectl first"
/>

## Configuration

<Inputs id="service-config">
```yaml
variables:
  - name: ServiceName
    type: string
    description: Name for your service
  - name: Environment
    type: enum
    options: [dev, staging, production]
\```
</Inputs>

## Deploy

<Command
    id="deploy"
    command="kubectl apply -f {{ .inputs.ServiceName }}-{{ .inputs.Environment }}.yaml"
    inputsId="service-config"
    title="Deploy Service"
    successMessage="Service deployed successfully!"
/>
```

## Developers actually self-serve

Most self-service tools fail because they're built for authors, not consumers. Runbooks flips that. Developers get a guided web UI with interactive forms, automated prerequisite checks, and step-by-step execution. They don't need to interpret a wall of markdown or figure out which flags to pass. The runbook does the thinking.

## Author in minutes, not weeks

No platform to deploy. No plugins to build. No YAML abstractions to learn. Write a runbook in your editor using MDX — markdown you already know, with a handful of interactive blocks. It lives in git, gets reviewed in PRs, and stays close to the infrastructure it documents.

## Open source and self-contained

Runbooks is a single Go binary. Download it, run it, done. No servers to maintain, no cloud accounts to configure, no dependencies to manage. Everything executes locally on the developer's machine.

Open source under the MIT License. Inspect the code, contribute, or fork it.
