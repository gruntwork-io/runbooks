## Authors write MDX

You write a runbook in MDX — markdown with interactive blocks — in your editor, just like code. Here's a real runbook:

```mdx
# Deploy a New Service

This runbook guides you through deploying a new
microservice to your infrastructure.

## Pre-flight checks

<Check
    id="check-node"
    path="checks/node-version.sh"
    title="Check Node.js Installation"
    successMessage="Node.js is installed and ready!"
    failMessage="Please install Node.js first"
/>

## Configure your service

<Inputs id="service-config">
  variables:
    - name: ServiceName
      type: string
      default: "my-service"
    - name: Environment
      type: enum
      options: [dev, staging, production]
    - name: Port
      type: int
      default: 3000
    - name: EnableMonitoring
      type: bool
      default: true
</Inputs>

## Deploy

<Command
    id="deploy"
    command="deploy {{ .inputs.ServiceName }}
      --env {{ .inputs.Environment }}"
    inputsId="service-config"
    title="Deploy the service"
/>
```

## Author in minutes, not weeks

No platform to deploy. No plugins to build. No YAML abstractions to learn. Write a runbook in your editor using MDX — markdown you already know, with a handful of interactive blocks. It lives in git, gets reviewed in PRs, and stays close to the infrastructure it documents.

<CardGrid>
  <Card title="MDX Authoring" icon="pen">
    Markdown + interactive blocks. If you can write a README, you can write a runbook.
  </Card>
  <Card title="Lives in Git" icon="github">
    Runbooks are code. Version them, review them, share them like any other file in your repo.

  <Card title="Watch Mode" icon="approve-check">

## Open source and self-contained

Gruntwork Runbooks is a single Go binary. Download it, run it, done. No servers to maintain, no cloud accounts to configure, no dependencies to manage. Everything executes locally on the developer's machine.

Open source under the MIT License. Inspect the code, contribute, or fork it.
